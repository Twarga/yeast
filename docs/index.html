Why it exists

Local VMs should feel like part of the project.

A useful VM workflow usually means downloading cloud images, creating qcow2 disks, writing cloud-init data, composing QEMU flags, tracking SSH ports, and remembering which files belong to which experiment.

Yeast makes that a project loop: describe the machine, pull the image, start it, SSH in, stop it, destroy it. The v0.1 release keeps the scope small so the core engine can become reliable before LabsBackery, MCP, provisioning, snapshots, and cloud workers build on top.

Current scope

Small core, serious foundation.

01

Project VMs

Each folder owns its own `yeast.yaml`, project identity, runtime state, disks, logs, and VM lifecycle.

02

Trusted Images

Yeast pulls supported Ubuntu cloud images into a shared cache and verifies checksums before use.

03

Cloud-init Bootstrap

VM startup includes generated cloud-init metadata, SSH keys, hostname, and predictable first-boot access.

04

Lifecycle Commands

`init`, `doctor`, `pull`, `up`, `status`, `ssh`, `down`, `destroy`, and `version` form the first working loop.

05

Human CLI

Terminal output is styled with Charm Lip Gloss so the CLI feels clean, readable, and modern.

06

Machine Output

JSON output stays plain and stable so future tools like LabsBackery and Yeast MCP can integrate without scraping terminal text.

Install

One script for Linux hosts.

The installer detects common package managers, prepares QEMU/KVM dependencies, builds Yeast, creates local cache folders, and runs `yeast doctor`. 

curl -fsSL https://raw.githubusercontent.com/Twarga/yeast/main/install.sh | bash
mkdir yeast-demo
cd yeast-demo
yeast init
yeast doctor
yeast pull ubuntu-24.04
yeast up
yeast status
yeast ssh web
