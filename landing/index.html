How it works

Short path from install to SSH.

Yeast keeps the loop tight: install, initialize, describe the lab, boot it, and connect to the running machine.

01

Create a project

Start in a normal folder. Yeast keeps project metadata and runtime state beside the lab itself.

mkdir my-lab && cd my-lab
02

Describe the machines

Use yeast init and edit yeast.yaml to define the VMs, disks, and networking.

yeast init
03

Boot with one command

yeast up validates the config, downloads trusted images if needed, starts QEMU/KVM, and runs cloud-init.

yeast up
04

SSH into a real VM

Once the guest is ready, use yeast ssh web-01 and keep working inside Linux.

yeast ssh web-01
Key features

Everything in Yeast v1.1 stays focused on the local engine.

Yeast ships the pieces that matter for repeatable local labs: project-local config, trusted image flow, first-boot provisioning, snapshots, private networking, and machine-readable output.

01 / Project-local

YAML-driven configuration

Describe the lab once in yeast.yaml and keep the whole setup beside the code it supports.

Version control friendly
02 / Auto-download

Trusted images on demand

yeast up downloads supported cloud images automatically when they are missing. yeast pull is optional.

Cache warming and browsing remain available
03 / First boot

cloud-init provisioning

Provision users, SSH keys, packages, files, and shell steps as the guest boots for the first time.

Repeatable setup
04 / Recovery

Snapshots and restore

Save a known-good state, test something risky, and jump back to the clean machine when you need to.

Stopped-VM snapshots
05 / Networking

Private lab networking

Build multi-VM labs with one project-level network and predictable per-instance IPv4 addresses.

Good for app, DB, and proxy topologies
06 / Automation

Stable JSON and events

Consume a versioned JSON contract and JSON Lines lifecycle events from tooling, scripts, and agents.

Machine-readable by design
